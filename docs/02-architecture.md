## Кто с кем «разговаривает»

Важнейший принцип: **инициатор связи — всегда плеер, а не сервер.**

Плеер сам, по своему таймеру (по умолчанию каждые ~5 минут — «collect interval»):

1. Стучится на сервер: «что мне положено показывать?»
2. Сервер отвечает списком нужных файлов и раскладок.
3. Плеер скачивает то, чего у него ещё нет, в локальный кэш.
4. Плеер показывает контент **из локального кэша**.

Отдельно есть **XMR** (порт 9505): он нужен, чтобы сервер мог «толкнуть» плееру
команду немедленно, не дожидаясь следующего опроса (например, кнопка «Collect Now»
— «обнови сейчас»). Это ускоряет доставку, но не обязательно для базовой работы.

### Почему это важно

- **Плеерам не нужен «белый» IP и проброс портов.** Они сами ходят наружу, как
  обычный браузер. Достаточно, чтобы на точке был обычный интернет.
- **Обрыв сети не гасит экран.** Видео играет из локальной копии; при
  восстановлении связи плеер догоняет обновления.
- **«Виден» наружу только сервер.** Только сервер CMS должен быть доступен
  плеерам по сети (по HTTP/HTTPS и порту XMR).

## Что где хранится (на сервере)

Всё состояние сервера лежит в папке `shared/` рядом с `docker-compose.yml`:

| Путь | Что там |
|------|---------|
| `shared/db/` | База данных MySQL: экраны, группы, расписания, пользователи |
| `shared/cms/library/` | **Загруженные видеофайлы** (самое ценное по объёму) |
| `shared/backup/` | Автоматические дампы БД, которые делает сам Xibo |
| `shared/cms/custom/`, `.../theme/custom/` | Кастомизации (обычно пусто) |
| `config.env` | Пароли и настройки (⚠️ секрет, не коммитить) |

👉 Отсюда вытекает стратегия бэкапа: сохранять `shared/db` (или дамп из
`shared/backup`), `shared/cms/library`, `config.env` и `*.yml`. Подробно —
[10-maintenance-backup.md](10-maintenance-backup.md).

## Разделение контента: как это работает логически

```
        РАСПИСАНИЕ (Schedule)
        ─────────────────────
  «Ролик А»  → группа «Все»       приоритет 0  (фон, по умолчанию)
  «Ролик Б»  → группа «Детские»   приоритет 5  (перекрывает фон)
  «Ролик В»  → группа «Стоматология» приоритет 5

        РЕЗУЛЬТАТ на конкретном телевизоре:
        ─────────────────────────────────
  ТВ в регистратуре      (только в «Все»)          → показывает А
  ТВ в детском отделении (в «Все» + «Детские»)     → показывает Б
  ТВ в стоматологии      (в «Все» + «Стоматология»)→ показывает В
```

Один телевизор может входить в несколько групп. Xibo показывает раскладку с
**наивысшим приоритетом** среди назначенных этому экрану. Так и получается
«всем — общее, отдельным — своё». Пошагово с картинками панели:
[08-groups-and-scheduling.md](08-groups-and-scheduling.md).

## Требования к серверу

| Параметр | Минимум | Рекомендуется |
|----------|---------|---------------|
| ОС | Linux (Ubuntu 22.04+ / Debian 12+) | Ubuntu 24.04 LTS |
| CPU | 2 ядра | 2–4 ядра |
| RAM | 2 ГБ | 4 ГБ |
| Диск | 20 ГБ | 50–100 ГБ (зависит от объёма видео) |
| Docker | Docker Engine + плагин Compose | последняя стабильная версия |
| Сеть | статический публичный IP или доменное имя | + HTTPS-сертификат |

Установка сервера — [03-server-install.md](03-server-install.md).
