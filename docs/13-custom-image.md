# 13. Свой образ плеера: Raspberry Pi и balenaOS (полный контроль)

Этот документ — для тех, кто хочет **полностью контролировать софт на точке**:
собрать один образ, прошить его на устройство и клонировать на всю сеть, а потом
обновлять всё «по воздуху». Это ответ на идею «накатить свою прошивку», но
сделанную правильно — **не на телевизоре, а на маленьком устройстве-плеере**,
которое для этого предназначено. Телевизор остаётся просто «монитором» по HDMI,
и его модель больше не важна.

> 📌 Это **альтернатива** штатному пути из [05-players-android-box.md](05-players-android-box.md).
> Для большинства точек приставка + приложение Xibo Player проще и быстрее.
> Берите этот путь осознанно — ради единообразия, контроля и OTA-обновлений.

---

## Сначала — честно про Raspberry Pi и Xibo

**Официального плеера Xibo для Raspberry Pi нет.** Плеер «Xibo for Linux»
рассчитан на x86-64 (Intel/AMD) и на ARM-процессоре Raspberry Pi не запускается.
Это официальная позиция Xibo — не тратьте время на попытки поставить туда обычный
Linux-плеер.

Поэтому на Pi есть два реальных пути:

| Путь | Что это | Плюсы | Минусы |
|------|---------|-------|--------|
| **Arexibo / ArexiboOS** | Неофициальный плеер Xibo на Rust для Linux/ARM, есть готовый образ для Pi | Лёгкий, работает на Pi (вкл. Pi 5), «почти готовый образ» | Неофициальный, поддерживается нерегулярно (последние правки ~май 2025) |
| **Android на Pi + офиц. приложение** | LineageOS/emteria на Pi, сверху штатный Xibo Player для Android | Официальный плеер, знакомая настройка | Android на Pi тяжелее, сложнее в сборке |

Отдельно стоит **balenaOS** — это не плеер, а операционная система для управления
**флотом** устройств: она запускает ваши Docker-контейнеры и обновляет их по
воздуху. В контейнер вы кладёте плеер (Arexibo или браузерный/kiosk-вариант).
Готового официального «Xibo-блока» под balena нет — плеерный контейнер вы
собираете сами. Зато получаете максимум контроля и честный OTA.

## Вариант A. Raspberry Pi + ArexiboOS (самый быстрый «свой образ»)

Подходит, если хочется именно готовый образ под Xibo с минимумом возни.

### Что нужно
- Raspberry Pi 4 или 5 (лучше 5, 2–4 ГБ RAM), корпус, питание, microSD 16–32 ГБ.
- HDMI-кабель к телевизору.
- Компьютер с программой **Raspberry Pi Imager** или **balenaEtcher** для записи
  образа на карту.

### Шаги
1. Скачайте образ **ArexiboOS** (проект Arexibo, репозиторий на GitHub) — это
   образ на базе Raspberry Pi OS 64-bit со встроенным плеером.
2. Запишите образ на microSD через Raspberry Pi Imager / balenaEtcher.
3. Вставьте карту в Pi, подключите HDMI и питание.
4. При первом запуске укажите плееру **адрес вашего CMS** и **CMS Secret Key**
   (как в [04-players-smart-tv.md](04-players-smart-tv.md), шаг 3) — через
   конфиг Arexibo или его экран настройки.
5. В веб-панели **Displays** авторизуйте новый экран.
6. Дальше — как любой плеер: положите экран в группы и назначьте контент
   ([08-groups-and-scheduling.md](08-groups-and-scheduling.md)).

### Клонирование на всю сеть
1. Настройте **один эталонный** Pi полностью (сеть, автозапуск, адрес CMS).
2. Снимите образ настроенной карты:
   - Linux/Mac: `sudo dd if=/dev/диск-карты of=etalon.img bs=4M status=progress`
   - или прочитайте карту через Win32DiskImager (Windows).
3. Записывайте `etalon.img` на остальные карты тем же Imager/Etcher.
4. Важно: **имя экрана (Display Name) должно быть уникальным** на каждой точке —
   поменяйте его после клонирования, иначе плееры «столкнутся» в панели.

> ⚠️ Arexibo — сообществный проект. Перед раскаткой на 50–200 точек прогоните его
> на 1–2 устройствах пару недель: проверьте стабильность, воспроизведение ваших
> роликов, поведение после отключения питания.

## Вариант B. Raspberry Pi + balenaOS (флот с OTA-обновлениями)

Это и есть «своя прошивка» в промышленном смысле: один образ на весь флот,
централизованные обновления по воздуху, мониторинг устройств из веб-панели. Плата
за это — больше первоначальной работы (нужно собрать контейнер с плеером).

### Как устроено balena
```
   Вы: git push  ──►  balenaCloud (сборка образа)
                          │  OTA (по воздуху)
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
     Pi #1             Pi #2             Pi #200
   balenaOS +        balenaOS +        balenaOS +
   контейнер-плеер   контейнер-плеер   контейнер-плеер
```
- **balenaOS** — минимальная ОС на устройстве, запускает Docker-контейнеры.
- **Fleet (флот)** — группа устройств с общим образом.
- **Release (релиз)** — новая версия контейнера; вы делаете `balena push`, и все
  устройства флота обновляются сами.
- **balenaCloud** — бесплатен до определённого числа устройств; для большого флота
  есть платные тарифы или self-hosted **openBalena**.

### Шаги (обзор)
1. Зарегистрируйтесь на **balenaCloud**, создайте **Fleet** под тип устройства
   (Raspberry Pi 4/5).
2. Скачайте из панели образ balenaOS для этого флота, запишите на microSD
   (balenaEtcher), впишите Wi-Fi/сеть при подготовке образа.
3. Загрузите Pi — устройство само появится в вашем флоте в панели balenaCloud.
4. Установите **balena CLI** на рабочий компьютер:
   ```bash
   npm install -g balena-cli
   balena login
   ```
5. Соберите контейнер-плеер. Минимальный `docker-compose.yml`/`Dockerfile` флота
   должен:
   - запускать графическую сессию (X/Wayland или прямой вывод в framebuffer);
   - стартовать плеер, указывающий на ваш CMS: либо **Arexibo** (собрать под ARM),
     либо браузер в режиме kiosk, открывающий выданную Xibo веб-раскладку.
6. Отправьте релиз во флот:
   ```bash
   balena push ИМЯ_ФЛОТА
   ```
   Все устройства скачают и применят обновление сами.
7. Настройки под конкретную точку (адрес CMS, имя экрана) задавайте через
   **переменные окружения устройства/флота** в balenaCloud — не пересобирая образ.

### Что это даёт
- **OTA**: поменяли плеер/настройки — один `balena push`, обновилась вся сеть.
- **Мониторинг**: онлайн/офлайн, логи, удалённый доступ к каждому устройству из
  браузера.
- **Единообразие**: все точки на одном, вами собранном образе.
- **Портфолио DevOps**: сборка образов, provisioning, fleet-management, OTA —
  сильный кейс в резюме.

### Цена подхода
- Больше первоначальной работы: контейнер-плеер вы собираете и поддерживаете сами.
- Если используете Arexibo внутри — та же оговорка про неофициальность.
- balenaCloud бесплатен до лимита устройств; дальше — платно или openBalena
  (свой сервер, больше администрирования).

## Что выбрать

| Ситуация | Рекомендация |
|----------|--------------|
| Нужно просто и быстро, парк смешанный | **Android-приставка + Xibo Player** ([05](05-players-android-box.md)) |
| Хочется готовый «свой образ» под Xibo на Pi | **Вариант A: ArexiboOS** |
| Нужен полный контроль + OTA на весь флот, готовы вложить труд | **Вариант B: balenaOS** |
| Нужен строго официальный плеер | Android-приставка/Smart TV (Pi официально не поддержан) |

Во всех вариантах **сервер CMS и логика групп/расписаний не меняются** — плеер
любого типа просто подключается к тому же Xibo CMS
([03-server-install.md](03-server-install.md)) и попадает в те же группи
([08-groups-and-scheduling.md](08-groups-and-scheduling.md)).
