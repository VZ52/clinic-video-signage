# 10. Обслуживание, бэкапы и обновление

Сервер работает сам, но раз в какое-то время его нужно резервировать и обновлять.
Этот документ — регламент, чтобы система пережила поломку диска, переезд на другой
сервер и обновление версии Xibo.

---

## Что именно надо бэкапить

Всё ценное состояние сервера лежит рядом с `docker-compose.yml`:

| Что | Путь | Зачем |
|-----|------|-------|
| База данных | дамп из `shared/backup/db/latest.sql.gz` **или** папка `shared/db` | экраны, группы, расписания, пользователи |
| Видеофайлы | `shared/cms/library/` | сами ролики (большой объём) |
| Настройки | `config.env` | пароли, параметры |
| Конфиг запуска | `docker-compose.yml`, прочие `*.yml` | как поднимать сервер |

> Xibo сам периодически делает дамп БД в `shared/backup/db/`. Файл
> `latest.sql.gz` — свежий снимок базы. Его достаточно для восстановления БД.

---

## Быстрый бэкап скриптом

В репозитории есть [`scripts/backup.sh`](../scripts/backup.sh). Он складывает БД,
библиотеку видео и конфиги в один архив с датой.

```bash
cd /opt/clinic-video-signage
./scripts/backup.sh                 # создаст backups/backup-ГГГГ-ММ-ДД.tar.gz
```

Восстановление — [`scripts/restore.sh`](../scripts/restore.sh):

```bash
./scripts/restore.sh backups/backup-2026-07-02.tar.gz
```

> ⚠️ Скрипты — вспомогательные. Перед боевым использованием проверьте
> восстановление на тестовом сервере: бэкап без проверенного восстановления —
> это не бэкап.

---

## Куда складывать бэкапы

Не храните бэкапы только на том же сервере — при отказе диска потеряете и данные,
и копию. Копируйте архивы:

- на другой сервер/облачное хранилище (`rsync`, `rclone` в S3/облако);
- периодичность: БД — ежедневно, видео — при изменениях (оно меняется реже).

Пример автоматизации через `cron` (ежедневно в 3:00):

```bash
crontab -e
# добавить строку:
0 3 * * * cd /opt/clinic-video-signage && ./scripts/backup.sh >> /var/log/xibo-backup.log 2>&1
```

## Обновление Xibo на новую версию

Xibo регулярно выпускает обновления (безопасность, новые функции). Порядок:

1. **Сделайте бэкап** (обязательно!): `./scripts/backup.sh`.
2. Посмотрите актуальную версию образа в официальном репозитории:
   https://github.com/xibosignage/xibo-docker/blob/master/docker-compose.yml
3. Обновите тег образа `cms-web` (и при необходимости `cms-xmr`, `mysql`) в вашем
   `docker-compose.yml` на новую версию.
4. Примените:

   ```bash
   cd /opt/clinic-video-signage
   docker compose pull          # скачать новые образы
   docker compose up -d         # пересоздать контейнеры
   docker compose logs -f cms-web   # дождаться завершения миграции БД
   ```

5. Проверьте панель и один-два экрана.

> ⚠️ Перескакивать сразу через несколько мажорных версий не рекомендуется — Xibo
> накатывает миграции БД. Обновляйтесь последовательно и сверяйтесь с
> [release notes Xibo](https://xibosignage.com/manual).
>
> ⚠️ Плееры тоже стоит держать в совместимой версии. Обычно свежий CMS работает и
> со слегка более старыми плеерами, но при мажорном обновлении обновите и
> приложение Xibo Player на устройствах.

## Плановый мониторинг

Раз в неделю бросайте взгляд на:

- **Displays** — все ли ключевые экраны онлайн? Кто давно не выходил на связь?
- Свободное место на диске сервера:

  ```bash
  df -h
  du -sh /opt/clinic-video-signage/shared/*
  ```

  Если `shared/cms/library` разрастается — удалите неиспользуемые старые ролики
  из **Library**.
- Живы ли контейнеры: `docker compose ps` (все `Up`).
- Есть ли свежий дамп в `shared/backup/db/`.

## Восстановление на новом сервере (переезд)

1. Установите Docker (см. [03-server-install.md](03-server-install.md), шаг 1).
2. Склонируйте репозиторий в `/opt/clinic-video-signage`.
3. Положите рядом сохранённый `config.env`.
4. Восстановите данные: `./scripts/restore.sh путь/к/бэкапу.tar.gz`.
5. `docker compose up -d`.
6. Проверьте, что адрес CMS в **Settings → Displays** совпадает с новым
   адресом/доменом (иначе плееры не найдут сервер). Если сервер сменил адрес —
   обновите его и на плеерах, либо заранее используйте доменное имя, а не IP
   (тогда достаточно перенаправить домен на новый сервер).

> 💡 Именно поэтому лучше с самого начала давать плеерам **доменное имя** CMS, а
> не голый IP: при переезде меняете только DNS, а не настройки на всех
> телевизорах.

---

## Безопасность (минимум)

- Смените дефолтный пароль `xibo_admin` (это делается при первом входе).
- Включите **HTTPS** (см. [03-server-install.md](03-server-install.md), шаг 7).
- Ограничьте доступ к серверу по SSH (ключи вместо пароля, firewall).
- Не коммитьте `config.env` в git (он в `.gitignore`).
- Заведите отдельных пользователей CMS для тех, кто грузит контент, вместо общего
  админа (**Administration → Users**).
