# balena-контейнер: плеер Xibo (Arexibo) на Raspberry Pi

Черновик balena-проекта, который превращает Raspberry Pi в управляемый плеер
Xibo с обновлениями «по воздуху» (OTA) на весь флот. Плеер — сообществный
**Arexibo** (Rust), подключается к вашему **Xibo CMS** так же, как любой другой
экран.

Концепция и сравнение путей — в [../docs/13-custom-image.md](../docs/13-custom-image.md),
Вариант B.

---

## ⚠️ Статус: ЧЕРНОВИК, не проверен на железе

Этот проект написан по официальной документации Arexibo и balena, но **не
запускался на реальном Raspberry Pi**. Его нельзя раскатывать сразу на сеть.
Сначала — обкатка на 1–2 устройствах и правки по разделу «Что проверить» ниже.

Причина честной оговорки: сборка и запуск Qt6 WebEngine + X-сервера на ARM —
самое хрупкое место, и его нельзя подтвердить без физического устройства.

---

## Что внутри

```
balena/
├── docker-compose.yml        # описание флота: один сервис player + том данных
├── README.md                 # этот файл
└── player/
    ├── Dockerfile.template    # сборка Arexibo (Rust+Qt6) под тип устройства
    └── start.sh               # запуск X + регистрация/старт Arexibo
```

- **Том `arexibo-data`** хранит конфигурацию и кэш медиа Arexibo — переживает
  перезапуски и обновления, повторная авторизация не нужна.
- **Переменные `CMS_HOST` / `CMS_KEY`** задаются в панели balenaCloud, а не в
  файлах — секреты в git не попадают.

## Предварительно

1. Аккаунт **balenaCloud** (бесплатен до лимита устройств) или свой **openBalena**.
2. **balena CLI** на рабочем компьютере:
   ```bash
   npm install -g balena-cli
   balena login
   ```
3. Работающий **Xibo CMS** (см. [../docs/03-server-install.md](../docs/03-server-install.md))
   и его **CMS Secret Key** (Administration → Settings → Displays).
4. Raspberry Pi 4/5 (64-bit), microSD, HDMI к телевизору.

## Развёртывание

### 1. Создать флот и завести устройство
- В balenaCloud создайте **Fleet** под ваш тип устройства (напр. Raspberry Pi 5).
- Скачайте образ balenaOS для этого флота, пропишите Wi-Fi/сеть, запишите на
  microSD (**balenaEtcher**), загрузите Pi. Устройство появится во флоте.

### 2. Задать переменные (секреты — только здесь, не в файлах)
В balenaCloud → Fleet (или конкретное устройство) → **Variables** добавьте:

| Переменная | Значение | Обязательна |
|-----------|----------|-------------|
| `CMS_HOST` | адрес CMS со слэшем, напр. `https://cms.вашдомен.ru/` | да |
| `CMS_KEY`  | CMS Secret Key из Xibo | да |
| `DISPLAY_ID` | фиксированный ID экрана (обычно не нужен) | нет |

### 3. Запушить код во флот
```bash
cd balena
balena push ИМЯ_ВАШЕГО_ФЛОТА
```
balena соберёт образ и раскатит на все устройства флота.

### 4. Авторизовать экран в Xibo
Плеер зарегистрируется и будет ждать. В панели Xibo: **Displays → Authorise**
(как в [../docs/04-players-smart-tv.md](../docs/04-players-smart-tv.md), шаг 4).
Окно плеера, по документации Arexibo, появляется **только после** авторизации.

### 5. Дать контент
Экран дальше — обычный: положите его в группы и назначьте раскладку
([../docs/08-groups-and-scheduling.md](../docs/08-groups-and-scheduling.md)).

### Обновления флота
Поменяли Dockerfile/скрипт → `balena push ИМЯ_ФЛОТА` → все устройства обновятся
сами. Настройки точки меняются через переменные, без пересборки.

## Имя экрана (Display Name)

Arexibo генерирует ID экрана из характеристик машины; в Xibo экран удобно
переименовать вручную (Displays → Edit → Name), например в
`Клиника-Центр — регистратура`. Чтобы отличать устройства во флоте, задавайте
понятные **имена устройств в balenaCloud** — это упростит сопоставление.

## Что проверить на реальном устройстве (перед раскаткой)

Отмечено как `TODO(verify)` в файлах:

1. **Сборка Qt6/QtWebEngine под ARM.** Точные имена пакетов Qt6 в Debian bookworm
   могут отличаться; сборка тяжёлая — при нехватке RAM добавьте swap или собирайте
   на мощной машине. Файл: `player/Dockerfile.template`.
2. **Запуск X-сервера.** `Xorg :0 ... vt1` может потребовать драйвера
   (`modesetting`/`fbdev`) или запуска через `startx`/`xinit`. При ошибке смотрите
   `/var/log/xorg.log` внутри контейнера. Файл: `player/start.sh`.
3. **Вывод видео и GPU-ускорение.** Проверьте плавность воспроизведения ваших
   роликов; возможно, понадобится включить KMS/GPU-настройки устройства.
4. **Автовосстановление.** Выдернуть-воткнуть питание: устройство должно само
   вернуться к показу (`restart: always` + том данных должны это обеспечить —
   подтвердите на практике).
5. **Размер образа и время первого деплоя.** Сборка из исходников на ARM долгая;
   учитывайте это при первой раскатке.

Диагностика: **balenaCloud → устройство → Logs / Terminal** — смотрите вывод
контейнера `player` (Arexibo пишет логи в stdout).

## Если это слишком сложно

Промышленный OTA-флот — не обязательный путь. Для большинства точек проще и
надёжнее **Android-приставка + официальное приложение Xibo Player**
([../docs/05-players-android-box.md](../docs/05-players-android-box.md)). Этот
balena-вариант берите, когда действительно нужен полный контроль над софтом и
централизованные обновления всего парка.
